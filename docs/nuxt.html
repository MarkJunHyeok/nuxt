Nuxt3를 능숙하게 사용하기 위해 기초부터 철저한 해설

본 문서에서는 Vue.js의 풀 스택 프레임워크인 Nuxt3 를 사용해 애플리케이션의 개발을 효율적으로 실시하고 싶다고 하는 사람을 대상으로 Nuxt3 에 대한 해설을 가능한 한 간단한 코드를 이용하면서 실시하고 있습니다.Vue.js 의 Composition API 의 코드를 이해할 수 있는 것을 전제로 기술하고 있으므로 Vue.js 를 사용해 기술한 코드의 상세에 대해 해설은 실시하고 있지 않지만, 본 문서를 다 읽으면 Nuxt 3 의 전체상과 기초적인 이해를 깊게 할 수 있습니다.

Nuxt3 버전은 beta 버전이나 rc판이 아닌 stable 3.0 을 이용하여 합니다.

※ 기사는 조금 길지만 기사는 현재도 업데이트 중입니다.

Nuxt3란

Nuxt는 Vue.js를 기반으로 개발된 풀스택의 meta 프레임워크입니다.
Vue.js는 주로 자바스크립트와 DOM을 이용함으로써 UI(User Interface)를 담당하지만 Nuxt에서는 UI 뿐만 아니라 SSR(서버 사이드 렌더링)을 포함한 렌더링 기능이나 메타태그, 라우팅, 오류 핸들링 등 본격적인 애플리케이션 구축에 필수적인 다양한 기능을 사전에 통합함으로써 애플리케이션 개발을 효율적으로 수행할 수 있습니다.예를 들어 라우팅이라면 Vue-Router가 사전에 내장되어 있을 뿐만 아니라 정해진 디렉토리에 파일을 작성하는 것만으로 자동으로 라우팅을 설정해 줍니다(파일베이스 라우팅). 또한 컴포넌트를 이용할 때 import문을 이용하지 않고도 자동으로 import를 수행하는 Auto-Import 기능도 가지고 있습니다.TypeScript 또한 지원하여 기본적으로 이용할 수 있습니다.이것은 일례이지만 이 외에도 다양한 기능이 포함되어 있으므로 본 문서에서 확인을 실시하겠습니다.

기본 디렉토리 구성

인스톨 직후의 디렉토리 구성을 확인해 봅시다.에디터는 VSCode를 이용하여 확인하고 있습니다.왼쪽 디렉토리 목록을 확인하면 node_modules 이외의 디렉토리는 없고 매우 심플한 구성임을 알 수 있습니다.브라우저 상에 표시되는 내용과 관련된 파일은 app.vue 파일뿐이라는 것을 알 수 있습니다.

app.vue파일

app.vue 파일의 내용을 확인하면 NuxtWelcome 태그가 설정되어 있는 것을 알 수 있지만 import문도 없기 때문에 어떤 파일이 NuxtWelecome 컴포넌트를 지원하는지는 알 수 없습니다.

NuxtWelcome 컴포넌트에 대한 설명은 문서에 기재되어 있으므로 확인합니다.문서에 따르면 NuxtWelcome 컴포넌트는 @nuxt/ui에 포함되어 있음을 알 수 있습니다.

@nuxt/ui 라이브러리의 내용물을 보면 https://github.com/nuxt/ui/tree/main/packages/templates/templates/welcome 의 index.html 파일로 초기화면에 표시되어 있는 것을 확인할 수 있으므로 내용물이 궁금하신 분들은 꼭 확인해 보시기 바랍니다.

초기화면에 표시된 내용을 변경하기 위해서는 초기화면에 표시되어 있던 메시지 "Remove this welcome page by replacing in app.vue with your own code."를 참고합니다.

app.vue 파일에서 NuxtWelcome 컴포넌트를 삭제하고 "Hello World"를 기술합니다.


<template>
  <h1>Hello World</h1>
</template>

브라우저에서 확인하면 "Hello World"를 확인할 수 있습니다.

app.vue 파일은 Nuxt3의 메인 컴포넌트입니다.여러 페이지를 작성할 경우 pages 디렉토리를 이용하지만 pages 디렉토리는 옵션이므로 반드시 이용할 필요는 없습니다.랜딩 페이지만 라우팅이 필요 없는 애플리케이션의 경우 app.vue 파일을 이용하는 것만으로 애플리케이션을 구축할 수 있습니다. app.vue 파일만 이용한 경우에는 라우팅의 vue-router를 번들할 필요가 없기 때문에 빌드 후 번들 파일 크기를 줄일 수 있습니다.번들 크기가 작아짐으로써 클라이언트가 받는 데이터의 양도 줄어들기 때문에 페이지의 표시 속도의 고속화로 이어집니다.

app.vue 이외의 파일

app.vue 파일 이외에 nuxt.config.ts, tsconfig.json, README.md , package.json 파일 등이 있습니다.nuxt.ocnfig.ts 는 Nuxt 에 관한 설정을 실시할 수 있는 파일로 본 문서에서도 nuxt.config.ts 파일을 이용해 설정을 실시합니다.확장자가 ts로 되어 있는 대로 디폴트로부터 TypeScript 에 대응하고 있습니다.tsconfig.json은 TypeScript 의 일반적인 설정 파일입니다.README.md 에 인스톨 커맨드등의 설명이 기술되어 있습니다.

디렉토리 구성

Nuxt 3 에서는 프로젝트 작성시에 node_modules 이외의 디렉토리는 존재하지 않지만 각각 다른 역할을 가지고 있는 디렉토리를 추가해 나가는 것으로 애플리케이션을 구축해 갑니다.디렉토리 이름은 Nuxt 3 중 사전에 정해져 있으며 컴포넌트를 이용할 경우 components 디렉토리를 작성합니다.components 디렉토리 아래에 컴포넌트 파일을 저장함으로써 Auto Imports 기능을 통해 import 문을 이용하지 않고도 자동으로 컴포넌트 import가 이루어집니다.그렇기 때문에 디렉토리 이름이나 작성한 디렉토리가 어떤 역할을 하고 있는지 디렉토리마다 이해해 둘 필요가 있습니다.

Auto Imports 기능을 이용하지 않고 import 문을 이용하여 컴포넌트를 가져올 수 있습니다.

pages 디렉토리

app.vue 파일만으로 애플리케이션을 구축할 수 있다는 것은 설명했지만 Nuxt를 이용하는 개발자는 보통 여러 페이지로 구성된 애플리케이션을 구축합니다.페이지를 추가하려면 pages 디렉토리를 이용합니다.Nuxt에서는 파일 베이스 라우팅을 채택하고 있으므로 pages 디렉토리 안에 파일을 작성하면 자동으로 라우팅이 설정됩니다.

pages 디렉토리를 작성하고 그 아래에 index.vue 파일을 작성하여 아래를 기술하십시오.


<template>
  <h1>Main Page</h1>
</template>
pages 디렉토리와 페이지 파일 작성은 npx nuxidd page index 명령어를 이용해도 작성할 수 있습니다.실행하면 pages 폴더와 그 안에 index.vue 파일이 생성됩니다.

pages/index.vue 파일을 작성했다고 브라우저 상의 표시가 바뀌지 않아야 합니다.

pages/index.vue 파일의 내용을 보여주기 위해서는 app.vue 파일에 NuxtPage 컴포넌트를 추가해야 합니다.


<template>
  <div>
    <NuxtPage />
  </div>
</template>

NuxtPage 태그를 추가하면 NextPage 태그를 추가한 곳에 pages/index.vue 파일의 컨텐츠가 표시됩니다.설명대로 설정했음에도 컨텐츠가 표시되지 않고 화면이 새하얗다면 npm run dev 명령을 다시 실행해 주세요.

또한 pages 디렉토리에 about.vue 파일을 작성하여 아래 내용을 기술합니다.


<template>
  <h1>About Page</h1>
</template>

about.vue 파일을 작성하면 브라우저의 URL에 localhost:3000/about을 직접 입력하고 접속하면 About Page가 나타납니다.자동으로 라우팅이 설정되어 있는 것을 알 수 있습니다.만약 Nuxt 에 자동 라우팅 기능이 갖추어져 있지 않다면 라우팅 파일을 이용하여 /about 에 액세스가 있을 경우 보여줄 컴포넌트가 무엇인지 지정해야 합니다.

pages 디렉토리에 존재하지 않는 URL에 액세스 한 경우는 404페이지가 표시됩니다.

pages 디렉토리에 파일을 작성하면 라우팅이 자동으로 설정된다는 점, 콘텐츠를 표시하기 위해서는 메인 파일인 app.vue 파일에서의 NuxtPage 태그가 필요하다는 것을 알았습니다.

수동 라우팅을 수행한 경험이 없는 사람은 아래 기사가 참고가 됩니다.보통은 어떻게 라우팅을 설정하는지 이해할 수 있습니다.

layouts 디렉토리

index.vue 파일, about.vue 파일의 페이지 사이를 이동할 수 있는 네비게이션을 추가하려면 두 파일에 동일한 내용을 추가하는 것은 비효율적입니다.레이아웃 파일을 작성함으로써 페이지에서 공통되는 내용을 하나의 파일에 기술할 수 있으며 레이아웃 파일을 공유하는 모든 페이지에 적용할 수 있습니다.

레이아웃을 이용하려면 layouts 디렉토리를 작성하여 default.vue 파일을 작성하십시오.<slot /> 은 필수 입니다.


<template>
  <div>
    <nav>ここにナビゲーションバーを入れる</nav>
    <slot />
  </div>
</template>

layouts 폴더에 default.vue 파일을 만드는 것만으로는 아무런 변화가 없습니다. app.vue 파일에 NuxtLayout 태그를 추가하고 NuxtPage 태그를 랩합니다.


<template>
  <NuxtLayout>
    <NuxtPage />
  </NuxtLayout>
</template>

app.vue 파일에 NuxtLayout 태그를 추가하면 자동으로 layouts 디렉토리에 존재하는 default.vue 파일의 내용이 적용됩니다.

http://localhost:3000/ 뿐만 아니라 http://localhost:3000/about 페이지에 들어가도 default.vue 파일의 내용이 적용되어 있음을 확인할 수 있습니다.

만약 레이아웃을 어떤 특정 페이지에서 적용하고 싶지 않은 경우에는 define PageMeta 함수로 제어할 수 있습니다.pages/about.vue 파일에 레이아웃을 적용하고 싶지 않다면 definePageMega 함수 인수에 layout 속성을 가진 객체를 추가하고 layout 값을 false로 설정합니다.설정하면 about.vue 파일만 레이아웃 적용이 되지 않습니다.index.vue 파일에 대해서는 default.vue 레이아웃이 적용된 상태로 유지됩니다.


<template>
  <h1>About Page</h1>
</template>
<script setup>
definePageMeta({
  layout: false,
});
</script>

layouts 폴더에 custom.vue 파일을 만듭니다.default와 달리 임의의 이름을 붙일 수 있습니다.


<template>
  <div>
    <nav>カスタムレイアウト</nav>
    <slot />
  </div>
</template>

추가한 custome.vue 파일 레이아웃을 이용하고자 하는 경우에도 definePageMeta 함수를 이용합니다.인수 개체의 layout 속성 값에 layouts 디렉토리에 있는 파일 이름을 지정합니다.여기서는 작성한 custom을 지정합니다.


<template>
  <h1>About Page</h1>
</template>
<script setup>
definePageMeta({
  layout: 'custom',
});
</script>

about 페이지에는 custom.vue 파일 레이아웃이 적용됩니다.

전체적으로 기본 설정할 레이아웃 파일을 default.vue에서 custom.vue 파일로 변경하고자 하는 경우에는 NuxtLayout 태그의 name props를 이용할 수 있습니다.


<template>
  <NuxtLayout name="custom">
    <NuxtPage />
  </NuxtLayout>
</template>

"/"에 접속하면 layouts/default.vue 파일 레이아웃이 아닌 layouts/custom 레이아웃이 적용됩니다.

v-bind 를 nameProps 를 이용하여 동적으로 레이아웃을 변경할 수도 있습니다.admin이 true일 경우 custom.vue 파일, admin이 false일 경우 default.vue 파일이 적용됩니다.


<template>
  <NuxtLayout :name="layout">
    <NuxtPage />
  </NuxtLayout>
</template>
<script setup>
const admin = true;
const layout = admin ? 'custom' : 'default';
</script>

레이아웃 파일 설정에 대해서는 about.vue 파일에 직접 NuxtLayout 컴포넌트를 이용할 수도 있습니다.name props에 레이아웃 파일명을 설정할 수 있는데 여기서는 name props를 설정하지 않았기 때문에 기본 레이아웃이 적용됩니다.


<template>
  <div>
    <NuxtLayout>
      <h1>About Page</h1>
    </NuxtLayout>
  </div>
</template>
page 파일에서 Nuxt Layout을 이용할 경우 Nuxt Layout 태그가 루트 엘리먼트가 되지 않도록 설정해야 합니다.pages/about.vue 파일의 루트 엘리먼트는 div로 하고 있습니다.

app.vue 파일에도 NuxtLayout 태그가 이용되고 있는지 확인해 둡니다.


<template>
  <NuxtLayout>
    <NuxtPage />
  </NuxtLayout>
</template>

브라우저에서 /about에 확인하면 2회 기본 레이아웃이 적용됩니다.

about.vue의 NuxtLayout에 name props를 추가하고 custom 레이아웃 설정만 설정한 후 custom 레이아웃만 반영하고 싶다면 definePageMeta 함수에서 layout을 false로 설정합니다. app.vue 파일에 적용되는 default 레이아웃은 적용되지 않고 custom 레이아웃만 적용됩니다.


<template>
  <div>
    <NuxtLayout name="custom">
      <h1>About Page</h1>
    </NuxtLayout>
  </div>
</template>
<script setup>

</script>

app.vue 파일은 Nuxt3의 메인 컴포넌트라고 설명했지만 app.vue는 필수가 아니기 때문에 삭제할 수 있습니다.삭제 진행 시 동작을 확인합니다.

pages 디렉토리의 index.vue 파일의 내용은 그대로입니다.


<template>
  <h1>Main Page</h1>
</template>

layouts 디렉토리에 default.vue 파일이 존재한 상태에서 브라우저에서 "/"에 접근하면 app.vue는 존재하지 않지만 default.vue 레이아웃이 적용됩니다.

레이아웃을 적용하고 싶지 않은 경우에는 definePageMeta 함수의 layout을 false로 설정하면 적용되지 않습니다.


<template>
  <h1>Main Page</h1>
</template>
<script setup>
definePageMeta({
  layout: false,
});
</script>

커스텀 레이아웃 custom.vue 파일의 레이아웃을 설정하고 싶다면 layout에 "custom"을 설정하는 것이 적용됩니다.


<template>
  <h1>Main Page</h1>
</template>
<script setup>
definePageMeta({
  layout: 'custom',
});
</script>

Nuxt Layout을 이용하여 레이아웃을 설정할 수 있는데, 이 경우 define PageMeta 함수에서 기본 레이아웃이 적용되지 않도록 설정해 두어야 합니다.


<template>
  <div>
    <NuxtLayout name="custom">
      <h1>Main Page</h1>
    </NuxtLayout>
  </div>
</template>
<script setup>
definePageMeta({

</script>

app.vue 파일을 필수가 아니기 때문에 삭제할 수 있다는 것, app.vue 파일이 존재하지 않는 경우에는 자동으로 기본 레이아웃이 적용된다는 것을 알았습니다.

동적으로 Layout 변경

페이지에 적용할 레이아웃을 동적으로 변경하고자 할 경우 setPageLayout 함수를 이용할 수도 있습니다.

처음에는 definePagaMeta 함수에서 layout을 false로 설정하고 있기 때문에 기본 레이아웃이 설정되어 있지 않은 상태입니다.화면상에 보여지는 "Update layout" 버튼을 클릭하면 custom 레이아웃이 적용됩니다.레이아웃 적용에 setPageLayout 함수를 이용하고 있습니다.


<template>
  <div>
    <button @click="enableCustomLayout">Update layout</button>
  </div>
</template>
<script setup>
function enableCustomLayout() {
  setPageLayout('custom');
}
definePageMeta({
  layout: false,
});
</script>

useRoute 함수를 이용하여 route.meta.layout 값을 설정하는 것으로도 동적으로 변경할 수 있습니다.


<template>
  <div>
    <button @click="enableCustomLayout">Update layout</button>
  </div>
</template>
<script setup>
const route = useRoute();
const enableCustomLayout = () => {
  route.meta.layout = 'custom';
};
definePageMeta({
  layout: false,
});
</script>

define PageMeta 함수를 이용하지 않을 경우 기본 layout이 적용되며 버튼을 클릭하면 custom 레이아웃이 적용됩니다.

이름이 붙은 slot 설정

app.vue 파일을 삭제하고 layouts 디렉토리에 default.vue 파일이 존재하며 index.vue 파일에 기본 레이아웃이 적용된 상태에서 slot 설정 동작을 확인합니다.


이름이 붙은 slot을 이용하기 위해 app.vue 파일을 삭제해야 하는 것은 아닙니다.

<template>
  <div>
    <nav>こにナビゲーションバーを入れる</nav>
    <slot />
  </div>
</template>


<template>
  <h1>Main Page</h1>
</template>

default.vue 파일에서 이름이 붙은 slot을 추가합니다.slot 태그에 name props를 추가하여 임의의 이름을 붙이고 있습니다.


<template>
  <div>
    <slot name="header" />
    <nav>こにナビゲーションバーを入れる</nav>
    <slot />
  </div>
</template>

index.vue 파일에서는 definePageMeta 함수에서 layout 속성을 false로 설정하고 NuxtLayout 컴포넌트를 추가합니다.name에는 default를 설정합니다.template 태그에 #header를 설정하고 태그 안에 이름붙인 slot을 설정한 위치에 표시하고 싶은 내용을 기술합니다.


<template>
  <div>
    <NuxtLayout name="default">
      <template #header>ヘッダー</template>
      <h1>Main Page</h1>
    </NuxtLayout>
  </div>
</template>

definePageMeta({
  layout: false,
});
</script>

template 태그에서 설정한 "헤더"가 표시되는 것을 알 수 있습니다.

Nuxt Layout에 명시적으로 name props에서 default를 설정하지 않으면 define PageMeta 함수에 의해 default 레이아웃이 적용되지 않으므로 name 설정이 필수입니다.또한 NuxtLayout 태그가 없는 경우 오류가 발생합니다.

components 디렉토리

app.vue 파일은 삭제하고 layouts 디렉토리에 default.vue 파일이 있는 상태에서 동작 확인을 실시하고 있습니다.


<template>
  <h1>Main Page</h1>
</template>;

재사용 가능한 컴포넌트 파일을 저장하기 위해 components 디렉토리를 작성합니다.

작성한 components 디렉토리에 Navbar.vue 파일을 작성합니다.Navbar.vue 파일은 default.vue 파일에서 네비테이션 바를 설정하기 위해 이용합니다.


<template>
  <nav>
    <a href="/">Home</a>
    <a href="/about">About</a>
  </nav>
</template>

작성한 Navbar 컴포넌트를 layouts 의 default.vue 파일에 추가합니다.Vue.js에서는 별도 파일에 기술한 컴포넌트를 이용하는 경우에는 import를 실시할 필요가 있습니다.Nuxt 3 에서는 components 디렉토리에 저장한 컴포넌트 파일은 자동으로 import 되기 때문에 import 처리를 할 필요가 없습니다.


<template>
  <div>
    <Navbar />
    <slot />
  </div>
</template>

설정 후에는 상단에 링크가 표시됩니다.Home, About 각각에 a태그로 링크가 부착되어 있으므로 클릭하면 페이지 이동을 할 수 있습니다.

a택은 추후 NuxtLink 태그로 변경하겠습니다.

Component Name

components 디렉토리에 새로 nav 디렉토리를 작성하여 방금 작성한 Navbar.vue 파일과 동일한 파일을 nav 디렉토리에 작성합니다.import 를 이용한 경우에는 컴포넌트 파일의 경로를 설정하기 때문에 같은 파일명이라도 문제 없습니다.Nuxt3에서는 자동으로 import를 해주는 경우에는 어떻게 파일을 구별할까요?nuxt.config.ts 파일로 설정을 하는 pathPrefix 의 값에 의해서 동작이 다릅니다.

pathPrefix 를 false 로 설정했을 경우의 동작 확인을 실시합니다.


export default defineNuxtConfig({
  devtools: { enabled: true },
  components: [
    {
      path: '~/components',
      pathPrefix: false,
    },
  ],
});

nav 디렉토리 아래에 작성했다 Navbar.vue 파일에는 이하의 코드를 기술합니다.


<template>
  <nav>
    <a href="/">Home2</a>
    <a href="/about">About2</a>
  </nav>
</template>

layouts/default.vue 파일 중 Navbar 태그를 이용합니다.


<template>
  <div>
    <Navbar />
    <slot />
  </div>
</template>

브라우저에서 확인하면 브라우저 상에는 components 디렉토리 바로 아래에 있는 Navbar 컴포넌트만 표시되고 터미널에는 WARNING이 표시됩니다.두 파일의 컴포넌트 이름이 중복됨을 알려줍니다.


WARN  Two component files resolving to the same name Navbar:

 - /Users/mac/Desktop/nuxt3-app/components/nav/Navbar.vue
 - /Users/mac/Desktop/nuxt3-app/components/Navbar.vue

components 디렉토리 바로 아래의 Navbar.vue 파일의 이름을 일시적으로 Navbar2.vue로 변경합니다.

변경 후 브라우저에서 확인하면 nav/Navbar 컴포넌트 파일의 내용이 표시됩니다.

pathPrefix 의 값이 false 인 경우 Navbar 파일이 하나라면 nav 디렉토리에 보존되어 있어도 Navbar 태그로 Auto imports 되는 것을 알 수 있습니다.

nuxt.config.ts 파일에서 pathPrefix 값을 삭제하면 nav 디렉토리에 저장했다 Navbar.vue 파일을 이용할 수 없게 되어 아래의 WARNING이 표시됩니다.


[Vue warn]: Failed to resolve component: Navbar

디렉토리가 중첩된 컴포넌트 파일은 디렉토리의 이름과 파일의 이름을 이용하여 Auto Imports 할 수 있습니다.nav 디렉토리에 있는 Navbar.vue 파일을 이용하고자 하는 경우에는 NavNavbar 태그를 이용합니다.디렉토리 이름을 선두에 추가하고 있습니다.


<template>
  <div>
    <NavNavbar />
    <slot />
  </div>
</template>

브라우저에서 확인하시면 nav 디렉토리에 있는 Navbar.vue 파일의 내용이 표시됩니다.같은 파일명이 components 디렉토리에 존재하는 경우는 디렉토리명을 이용하는 것으로 식별할 수 있다는 것을 알았습니다.

파일명이 Navbar.vue가 아닌 NavBar.vue 파일이라고 한 경우는 NavNavBar 로도 nav 디렉토리 NavBar 를 AutoImports 해주지 않습니다.NavNavBar 에서는 nav/nav/Bar.vue 파일을 import 하고 있는 것입니다.

Auto Imports를 이용하지 않을 경우 일반 import를 이용할 수 있습니다.


<template>
  <div>
    <Navbar />
    <slot />
  </div>
</template>
<script setup>
import Navbar from '@/components/nav/Navbar';
</script>

ComponentsのLazy Loading

Nuxt의 컴포넌트는 Lazy Loading이라는 기능을 가지고 있기 때문에 필요할 때만 Lazy Loading을 설정한 컴포넌트에 대응하는 Java Script 파일을 다운로드 할 수 있습니다.Lazy Loading 기능을 이용하여 Java Script 번들 크기를 줄일 수 있습니다.

컴포넌트의 Lazy Loading을 이용했을 때와 하지 않을 때의 동작을 네트워크 탭을 보면서 확인합니다.

components 폴더에 Coupon.vue 파일을 만듭니다.코드의 내용물을 아래와 같습니다.


<template>
  <div>Couponコード:1234</div>
</template>

ref 함수를 이용하여 정의한 show 변수와 v-if 디렉토리를 이용하여 페이지가 표시되었을 때는 Coupon 컴포넌트는 숨기고 버튼을 클릭했을 때 표시할 수 있도록 설정합니다.


<template>
  <div>
    <h1>Main Page</h1>
    <button @click="handleClick">Coupon Get</button>
    <Coupon v-if="show" />
  </div>
</template>
<script setup>
const show = ref(false);

const handleClick = () => {
  show.value = true;
};
</script>

개발자 도구의 네트워크 탭을 열어 페이지를 표시합니다.일련의 다운로드 파일 중에 Coupon.vue 파일이 포함되어 있습니다.

버튼을 클릭하면 Coupon.vue 파일의 내용이 표시됩니다.

그런 다음 Lazy Loading을 확인하기 위해 Coupon 앞에 Lazy를 추가한 태그로 변경합니다.


<template>
  <div>
    <h1>Main Page</h1>
    <button @click="handleClick">Coupon Get</button>
    <LazyCoupon v-if="show" />
  </div>
</template>
<script setup>
const show = ref(false);

const handleClick = () => {
  show.value = true;
};
</script>

페이지를 열고 네트워크 탭을 봐도 Copon.vue 파일을 찾을 수 없습니다.페이지 상의 버튼을 클릭하십시오.버튼을 클릭한 후 네트워크 탭을 보면 마지막에 Coupon.vue 파일이 추가되는 것을 알 수 있습니다.이렇게 Lazy Loading을 이용하면 컴포넌트가 필요할 때만 해당 컴포넌트와 관련된 Java Script 파일을 다운로드 받을 수 있습니다.

네비게이션 설정

Navbar.vue 파일 중에서는 페이지를 이동하기 위해 a태그를 이용하고 있었습니다.


<template>
  <nav>
    <a href="/">Home</a>
    <a href="/about">About</a>
  </nav>
</template>

a택을 이용해도 페이지를 이동할 수는 있지만 a 태그에서는 페이지를 이동할 때마다 페이지를 다시 불러오기가 이루어집니다.클릭할 때마다 페이지 전체를 읽기보다는 페이지 안에서 업데이트가 필요한 부분만 업데이트가 이루어지도록 a 태그에서 NuxtLink 컴포넌트를 이용하기 위해 NuxtLink 태그로 변경합니다.


<template>
  <nav>
    <NuxtLink to="/">Home</NuxtLink>
    <NuxtLink to="/about">About</NuxtLink>
  </nav>
</template>

a 태그에서 NuxtLink 컴포넌트로 변경하면 페이지 전체가 다시 읽히지 않기 때문에 페이지 이동이 원활하게 이루어집니다.디벨로퍼툴에서 확인하면 NuxtLink 태그를 이용한 장소에는 a 태그가 설정되어 있는 것을 확인할 수 있지만 Java Script 에 의해 본래의 a 태그와는 다른 동작이 되도록 제어되어 있습니다.NuxtLink 에서 to props 로 설정할 수 있는 값은 내부 링크 뿐만 아니라 외부 사이트를 지정할 수도 있습니다.외부 사이트의 경우 일반 a태그처럼 동작하며 내부 링크처럼 원활하게 페이지가 이동하지는 않지만 내부 링크와 외부 사이트 링크에서 태그를 구분해서 사용할 필요는 없습니다.또한 a 태그와 같이 target props(target 속성)를 설정할 수도 있습니다.


<template>
  <nav>
    <NuxtLink to="/">Home</NuxtLink>
    <NuxtLink to="/about">About</NuxtLink>
    <NuxtLink to="https://google.com" target="_blank">Google</NuxtLink>
  </nav>
</template>

URL 설정은 to props로 하고 있지만 to의 alias로서 href props도 이용 가능합니다.지금까지의 동작 확인에서 알 수 있듯이 Nuxt3로 링크를 설정하는 경우에는 a태그가 아닌 NuxtLink를 이용해 설정해 갑니다.

Nuxt3의 라우팅에는 Vue Router 라이브러리가 이용되고 있습니다.

assets 디렉토리 설정

assets 디렉토리에는 stylesheets나 fonts 등의 정보를 저장합니다.assets 디렉토리에 css나 이미지를 저장하고 assets 디렉토리를 이용했을 경우의 동작을 확인합니다.

CSS 파일 설정

assets 디렉토리에 css 디렉토리를 작성하여 style.css 파일을 작성합니다.

style.css 파일에 다음을 기술합니다.


h1 {
  color: gray;
}

css 파일은 Nuxt 설정 파일인 nuxt.config.ts 파일로 지정할 수 있습니다.


// https://nuxt.com/docs/api/configuration/nuxt-config
export default defineNuxtConfig({
  css: ['/assets/css/style.css'],
});

브라우저에서 확인하면 스타일이 적용된 것을 알 수 있습니다.

index.vue 파일에서 직접 index.css 파일을 import 할 수도 있습니다.


<template>
  <div>
    <h1>Main Page</h1>
  </div>
</template>
<script setup>
import '@/assets/css/style.css';
</script>

script 태그가 아닌 style 태그로도 import 할 수 있습니다.이 경우 import 전에 @를 붙여 진행합니다.scoped를 붙이면 적용 범위를 한정할 수 있습니다.scoped 는 필수가 아닙니다.


<template>
  <div>
    <h1>Main Page</h1>
  </div>
</template>
<script setup></script>
<style scoped>
@import '@/assets/css/style.css';
</style>

이미지 파일 저장

이미지 파일을 assets 디렉토리에 저장합니다.여기에서는 icon.png 파일을 assets 폴더 바로 아래에 저장하고 있습니다.

index.vue 파일에서 img 태그를 이용하여 assets 디렉토리의 icon.png 파일을 지정하는데 경로의 선두에는 "~"가 필요합니다.


<template>
  <div>
    <h1>Main Page</h1>
    <div>
      <img src="~/assets/icon.png" alt="Nuxt3 Icon" />
    </div>
  </div>
</template>

디벨로퍼툴의 요소를 확인해보면 src속성의 값은 /assets/icon.png이 아닌 선두에 /_nuxt/가 추가되어 있음을 알 수 있습니다.

이와 같이 assets 디렉토리를 이용한 css 파일이나 화상을 저장해 이용할 수 있지만 경로 설정 등에 주의가 필요합니다.

public 디렉토리 설정

public 디렉토리에는 이미지 파일과 같은 정적인 파일을 저장할 수 있습니다.public 디렉토리를 이용하기 위해 public 디렉토리를 작성합니다.

assets 디렉토리에서도 이미지를 저장하고 브라우저 상에서 표시할 수 있었지만 assets 디렉토리와 설정 방법이 다릅니다.

public 디렉토리에 icon.png 파일을 저장합니다.index.vue파일의 img태그에서 icon.png을 지정하는데 public은 필요하지 않습니다.


<template>
  <div>
    <h1>Main Page</h1>
    <div>
      <img src="/icon.png" alt="Nuxt3 Icon" />
    </div>
  </div>
</template>

브라우저에서 확인하면 지정한 이미지가 표시됩니다.경로는 img태그의 src로 설정한 /icon.png과 같은 값으로 되어 있는 것을 알 수 있습니다.

favicon 설정

public 디렉토리에 이미지 파일 설정 방법을 알았으니 favicon 설정 방법에 대해 확인해 두겠습니다.

Nuxt 3에서는 meta 태그 등의 기본 설정을 Nuxt 설정 파일인 nux.config.ts 파일로 할 수 있습니다.meta 태그에 대해서는 나중에 설명을 드리겠지만 title과 description만 설정을 하고 싶다면 아래와 같이 할 수 있습니다.설정은 모든 페이지에서 반영됩니다.


// https://nuxt.com/docs/api/configuration/nuxt-config
export default defineNuxtConfig({
  css: ['/assets/css/style.css'],
  app: {
    head: {
      title: 'Nuxt 3 basic',
      meta: [{ name: 'description', content: 'Nuxt 3 for beginners' }],
    },
  },
});

설정을 실시하면 브라우저 탭에 설정한 title의 값이 표시됩니다.

head 태그 중 meta 태그의 description에도 설정값이 반영되니 확인해주세요.

다음은 public 디렉토리에 저장한 favicon을 설정합니다.


// https://nuxt.com/docs/api/configuration/nuxt-config
export default defineNuxtConfig({
  css: ['/assets/css/style.css'],
  app: {
    head: {
      title: 'Nuxt 3 basic',
      meta: [{ name: 'description', content: 'Nuxt 3 for beginners' }],
      link: [{ rel: 'icon', href: '/icon.png' }],
    },
  },
});

브라우저를 확인하면 설정한 favicon이 브라우저 탭에 표시되는 것을 볼 수 있습니다.

브라우저 상에서는 아래와 같이 설정됩니다.Nuxt3의 문서 사이트와 동일하게 설정되어 있습니다.


<link rel="icon" href="/icon.png">

이미지 설정을 통해 public 디렉토리를 이용하는 방법을 이해할 수 있었습니다.

composables 디렉토리

composables 디렉토리 안에 composable한 함수를 포함하는 파일을 저장하면 컴포넌트에서 import 하지 않고(Auto imports) 이용할 수 있습니다.composables 함수는 reactive 변수와 로직을 컴포넌트에서 분리하여 재사용할 수 있도록 한 함수입니다.

composables 같은 함수가 어떤 것인지 확실하지 않은 사람도 있다고 생각하기 때문에 useCounter라는 이름의 composables한 함수를 작성하여 동작확인을 합니다.composables 함수는 Nuxt 특유의 함수가 아닌 Vue.js에서도 자주 이용합니다.다른 프레임워크에서 이용되고 있는 생각으로 React에서는 커스텀 훅에 대응합니다.

composables 디렉토리를 작성하여 useCounter.ts 파일을 만듭니다.useCounter 함수 인수에서 initialValue 를 받아 ref 함수의 초기값으로 count 변수를 정의하고 있습니다.count 변수를 업데이트하기 위한 inc, dec 함수를 정의하고 return 에서 count, inc, dec 을 되돌린다는 내용입니다.composables 한 함수에는 ref 함수로 정의된 reactive 변수와 로직만 포함되어 있습니다.


export function useCounter(initialValue: number) {
  const count = ref(initialValue);
  const inc = () => (count.value = count.value + 1);
  const dec = () => (count.value = count.value - 1);
  return {
    count,
    inc,
    dec,
  };
}

pages/index.vue 파일에서 useCounter를 이용하고자 하는 경우에는 아래와 같이 import 없음(Auto imports)으로 이용할 수 있습니다.


<template>
  <div>
    <h1>Main Page</h1>
    <div>Count:{{ count }}</div>
    <div>
      <button @click="() => inc()">increase</button>
      <button @click="() => dec()">decrease</button>
    </div>
  </div>
</template>
<script setup>
const { count, inc, dec } = useCounter(100);
</script>

브라우저 확인 시 useCounter 인수로 설정한 100이 표시되며 increase 버튼 또는 decrease 버튼을 누르면 숫자가 증감합니다.

plugins 디렉토리

plugins 디렉토리를 작성하고 플러그인 파일을 작성하면 앱케이션을 시작할 때 Nuxt가 작성한 플러그인을 자동으로 등록을 해주기 때문에 등록 작업을 할 필요가 없습니다.

plugins 디렉토리를 작성하여 가장 심플한 플러그인 hello.ts 를 이용하여 설정 방법을 확인합니다.

인수msg를 받고 Hello와 받은 인수를 표시하기만 하면 되는 플러그인입니다.


export default defineNuxtPlugin(() => {
  return {
    provide: {
      hello(msg: string) {
        return `Hello ${msg}!`;
      },
    },
  };
});

plugins 디렉토리에 플러그인을 작성하는 것만으로 컴포넌트부터 이용할 수 있습니다.index.vue에서 compsables의 useNuxtApp 함수를 실행하여 플러그인 $hello를 꺼냅니다.


<template>
  <div>
    <h1>Main Page</h1>
    <h2>{{ $hello('World') }}</h2>
  </div>
</template>

<script setup>
const { $hello } = useNuxtApp();
</script>

브라우저에는 플러그인 hello를 이용한 Hello World!가 표시됩니다.

라우팅 설정

pages 디렉토리의 index.vue, about.vue 파일을 작성하여 자동으로 라우팅이 설정되는 것을 확인하고 있는데 계층 페이지 등 라우팅 설정에 대해 확인해 둡니다.

계층 페이지 작성

About 페이지는 pages 디렉토리 바로 아래에 about.vue 파일을 작성하여 브라우저에서 URL 의 /about 로 접근함으로써 About 페이지의 내용을 표시할 수 있었습니다.다음은 URL 이 /users/list 와 같이 계층으로 되어 있는 경우의 라우팅 방법에 대해 확인합니다.

pages 디렉토리 안에 users 디렉토리를 작성하고 그 아래에 list.vue 파일을 작성하여 아래 코드를 기술합니다.


<template>
  <h1>User Listページ</h1>
</template>

Nuxt3가 자동으로 라우팅을 수행하므로 파일 작성 후 브라우저에서 /users/list에 접속하면 list.vue의 template 태그에 기술한 내용이 표시됩니다.이렇게 Nuxt를 이용하면 계층적인 URL 라우팅도 자동으로 해줍니다.

추가한 List 페이지의 라우팅을 Navbar.vue 파일에 추가합니다.NuxtLink의 to props에는 객체를 이용하여 name 속성에서 라우팅 이름을 이용하는 것으로도 설정할 수 있습니다.


<template>
  <nav>
    <NuxtLink to="/">Home</NuxtLink>
    <NuxtLink to="/about">About</NuxtLink>
    <NuxtLink :to="{ name: 'users-list' }">User List</NuxtLink>
  </nav>
</template>

users-list라는 이름이 갑자기 나왔는데 라우팅의 name 속성으로 설정할 값에 대해서는 Chrome이면 Extensions에 Vue Devtools를 넣어두면 아래와 같이 페이지 이름이 표시됩니다.

자동으로 라우팅이 설정되므로 파일 이름을 업데이트하면 name 값도 달라집니다.Vue Router를 수동으로 설정하는 경우에는 name을 설정하는 것으로 URL을 변경해도 name으로 링크를 설정하는 경우에는 각 파일에서의 URL 변경의 필요가 없어집니다.

링크 추가를 하였으므로 Home, About 페이지로의 이동과 같이 페이지 상단의 링크에서 User List 페이지로 이동할 수 있습니다.

/users/list에서 액세스를 할 수 있게 되었지만 /users/에 액세스하고 싶은 경우 페이지를 표시하고 싶다면 users 디렉토리 아래에 index.vue 파일을 작성합니다.


<template>
  <h1>User Indexページ</h1>
</template>

/users 에 접속하면 index.vue 에 기술한 내용이 표시됩니다.

페이지의 네스트화

pages 디렉토리 아래에 users 디렉토리와 같은 이름의 users.vue 파일을 작성합니다.


<template>
  <h1>Usersページ</h1>
</template>

users.vue 파일을 작성하면 /users, /users/list 어느 쪽에 접근하든 users.vue 파일에 기술한 Users 페이지만 표시됩니다.

users/list 컴포넌트의 내용을 보여주려면 부모 컴포넌트에 해당하는 users.vue 파일에서 NuxePage 태그를 추가해야 합니다.


<template>
  <div>
    <h1>Usersページ</h1>
    <NuxtPage />
  </div>
</template>

NuxtPage 태그를 추가 후에 /users/list에 접속하면 아래와 같이 users.vue 파일의 내용과 users/list.vue 파일의 내용이 표시됩니다.list.vue 파일의 내용은 NuxtPage 태그를 추가한 위치에 표시됩니다.

/users/에 접속하면 users.vue 파일의 내용과 /users/index.vue 파일의 내용이 표시됩니다.이렇게 페이지 네스트화를 할 수 있었습니다.

다이나믹 라우팅

/users/list 에 접속하면 User List 페이지가 나오는데, /users/1, /users/2와 같이 /users/아래가 동적으로 변하는 경우에 /users/아래 값을 파라미터로 받아 파라미터마다 다른 페이지 내용을 표시하고자 하는 경우 설정 방법을 확인합니다.파라미터는 동적으로 변하기 때문에 users 디렉토리에 [id].vue 파일을 작성합니다.[]로 둘러싼 값을 임의의 이름을 붙일 수 있습니다.


<template>
  <p>ユーザID</p>
</template>

/users/1, /users/2, /users/100 에 접속해도 같은 내용이 표시됩니다.

id에 들어간 값은 $route 객체를 이용하여 얻을 수 있습니다.$route Object의 params 안에 id 값이 들어 있습니다.


<template>
  <p>ユーザID: {{ $route.params.id }}</p>
</template>

/users/4 에 접속하면 id 의 4 가 브라우저 상에 표시됩니다.파라미터 값에 따라 페이지 내용을 바꿀 수 있게 되었습니다.

script 태그 중 id에 접근하기 위해 $route가 아닌 useRoute 함수를 이용할 수 있습니다.


<template>
  <p>ユーザID: {{ $route.params.id }}</p>
</template>
<script setup>
const router = useRoute();

console.log(router.params.id);
</script>

브라우저의 개발자 도구 콘솔에는 id 값이 표시됩니다.

Dynamic 라우팅을 이용할 때 [id].vue라는 파일명뿐만 아니라 user-[id].vue 파일이라는 이름을 붙일 수도 있습니다.id 부분이 동적으로 바뀌기 때문에 브라우저에서 접속할 경우에는 /users/user-4 형태로 접속하게 됩니다.user-는 고정값입니다.

middleware 디렉토리

middleware 디렉토리 아래에 파일을 작성하는 것으로 Route Middleware 설정을 실시할 수 있습니다.Route Middleware 를 이용하는 것으로 페이지간의 이동, 사이트에의 액세스를 실시하고 페이지가 표시되기 전에 미리 설정하고 있던 처리를 실시할 수 있습니다.예를 들어 어떤 특정 페이지에는 관리자 권한을 가지고 있는 사용자밖에 액세스시키고 싶지 않다고 하는 경우에 페이지를 표시하기 전에 인증 체크를 실시하는 것이 가능해집니다.인증의 체크에 실패했을 경우에는 액세스 한 페이지를 표시하지 않고 다른 페이지로 리디렉션시키는 등을 실시할 수 있습니다.

Route Middleware 동작 확인을 하기 위해서 middleware 디렉토리를 작성하고 그 아래에 auth.ts 파일을 작성합니다.

auth.ts 파일에서는 defineNuxtRouteMiddleware 함수를 이용합니다.defineNuxtRouteMiddleware 함수 안에서는 to, from 객체가 전달되기 때문에 어떤 값이 들어있는지 확인합니다.


export default defineNuxtRouteMiddleware((to, from) => {
  console.log('from', from);
  console.log('to', to);
});

설정한 middleware는 about.vue 파일의 definePageMeta 함수로 설정할 수 있습니다.middleware 속성 값에 작성한 파일 이름 auth를 설정합니다.


<template>
  <div>
    <h1>About Page</h1>
  </div>
</template>
<script setup>
definePageMeta({
  middleware: 'auth',
});
</script>

about 페이지 이외의 페이지에서 about 페이지에 접속하면 콘솔 로그에 from과 to 객체가 표시됩니다.여기서는 루트 페이지에서 about 페이지로 이동하고 있습니다.

from은 어느 페이지에서 이동하고 있는가? to는 어느 페이지로 이동하려고 하는지 알 수 있습니다.from, to 객체에는 각 페이지의 경로와 파라미터 등이 포함되어 있음을 알 수 있습니다.

middleware 와 from 오브젝트를 이용함으로써 / (루트)페이지에서 접속이 있을 경우에만 다시 / (루트)페이지로 리디렉션을 시킬 수도 있습니다.

from의 fullPath 값이 "/"인 경우에는 router의 helper 함수이다. navigateTo함수에서 "/"로 리디렉션시키도록 설정되어 있습니다."/"에서 /about 로 페이지 이동하면 about 페이지로 이동할 수 없습니다."/"이외 기타 페이지에서는 about 페이지로 이동할 수 있습니다.


export default defineNuxtRouteMiddleware((to, from) => {
  if (from.fullPath === '/') {
    return navigateTo('/');
  }
});

페이지 이동을 중지하고자 하는 경우에는 about Navigation 함수를 이용할 수 있습니다.


export default defineNuxtRouteMiddleware((to, from) => {
  if (from.fullPath === '/') {
    return abortNavigation();
  }
});

접근하는 사용자가 관리자인지 확인하는 함수를 만들어 middleware를 이용해 페이지 접근 제한을 걸 수 있습니다.관리자가 아닌 경우 /login으로 리디렉션되고 관리자인 경우 그대로 페이지가 표시됩니다.


export default defineNuxtRouteMiddleware((to, from) =>
  //isAdmin関数はアクセスしているユーザが管理者かどうかチェックする関数です
  if (isAdmin() === false) {
    return navigateTo('/login')
  }

Data Fetching

애플리케이션을 구축하는 경우에 Data Fetching에 의해 데이터를 취득해 취득한 데이터를 브라우저 상에 표시시킬 필요가 있습니다.Nuxt3에서는 데이터를 취득하기 위한 함수가 사전에 준비되어 있기 때문에 그 함수들을 이용할 수 있습니다.Nuxt 3 에서 이용할 수 있는 데이터 취득에 관한 함수는 useFetch, useLazyFetch, useAsyncData, useLazyAsyncData 이렇게 4가지입니다.각각의 이용 방법을 확인해 가겠습니다.

useFetch

useFetch는 Promise를 반환하는 함수로 GET 요청이면 인수에 URL을 지정하는 것만으로 지정한 URL에서 데이터를 얻을 수 있습니다.

posts 디렉토리를 작성하고 index.vue 파일을 작성하여 useFetch 함수를 이용하여 무료로 이용할 수 있는 JSONPlaceHolder에서 데이터를 가져옵니다.이용하는 URL은 'https://jsonplaceholder.typicode.com/posts/' 에서 접속하면 100건의 Post 데이터를 얻을 수 있습니다.

useFetch를 이용하여 어떤 데이터가 반환되는지 확인합니다.


<script setup>
const response = await useFetch('https://jsonplaceholder.typicode.com/posts/');
console.log(response);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
  </div>
</template>

/posts에 브라우저에서 접속하면 브라우저 개발자 도구 콘솔에는 useFetch 함수 실행에 따라 반환되는 정보가 표시됩니다.

반환된 데이터 중에는 data 외에 error, pending, execute, refresh가 포함되어 있음을 알 수 있습니다.

콘솔의 메시지 정보에서 data, error, pending 는 RefImp 로 표시되어 있는 것으로부터 리얼한 상태로 되돌리고 있는 것을 알 수 있습니다.ref 함수에서 변수를 정의했을 때와 같으므로 script 태그 내에서는 data.value 에서 data 에 포함된 정보에 접근할 수 있습니다.나머지 execute, refresh는 함수입니다.

data에는 취득한 데이터, error에는 에러 메시지(오류가 있는 경우), pending에는 데이터 취득 중인지 Boolean 값이 포함되어 있습니다.execute, refresh 는 함수이므로 실행할 수 있으며 실행하면 다시 데이터 취득을 할 수 있습니다.

data에는 100건의 Post 데이터가 포함되어 있으므로 분할 대입에서 data만 꺼내 data에는 알기 쉽게 posts라는 별칭을 붙여 v-for 디렉티브로 전개하고 브라우저 상에 제목을 표시시킵니다.


<script setup>
const { data: posts } = await useFetch(
  'https://jsonplaceholder.typicode.com/posts/'
);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

/posts에 접속하면 JSONPlace Holder에서 취득한 100건의 Post 정보가 표시됩니다.

components 디렉토리의 Navbar.vue 파일 네비게이션에 Post List도 추가해 둡니다.

data 뿐만 아니라 error의 내용도 확인합니다.error의 내용은 script 태그 내에서는 error.value에서 확인할 수 있습니다.


<script setup>
const { data: posts, error } = await useFetch(
  'https://jsonplaceholder.typicode.com/posts/'
);

</script>

useFetch의 처리가 정상적으로 이루어진 경우에는 error.value의 값은 null이 됩니다.

오류를 의도적으로 발생시키기 위해 useFetch 함수 인수를 존재하지 않는 URL로 변경하면 error.value에는 오류 메시지가 저장됩니다.v-if 디렉티브를 이용하여 error에 값이 포함되어 있거나 분기를 수행하고 error가 포함되어 있는 경우 브라우저 상에 오류 메시지를 표시하게 합니다.


<script setup>
const { data: posts, error } = await useFetch(
  'https://jsonplaceholder.typicode.com/post/'
);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

useFetch를 통한 요청이 실패한 경우 오류 메시지가 표시됩니다.

refresh 함수를 이용하여 데이터를 재취득하기 위해 재취득 버튼을 추가합니다.버튼을 클릭하면 refresh 함수가 실행되고 재취득이 이루어집니다.JSONPLace Holder 상의 데이터에 변경이 없기 때문에 재취득해도 화면상에는 변화가 없지만 디벨로퍼 툴의 네트워크 탭을 보면 요청이 전송되어 refresh 함수가 동작하고 있음을 확인할 수 있습니다.


<script setup>

  data: posts,
  error,
  refresh,
} = await useFetch('https://jsonplaceholder.typicode.com/posts/');

</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <button @click="refresh()">再取得</button>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

pending에 대해서는 useLazyFetch 함수 부분에서 확인합니다.

useAsyncData

useFetch 함수는 useAsyncData 함수와 $fetch 함수의 이용을 편리하게 한 래퍼입니다.따라서 useFetch 함수와 동일한 처리를 useAsyncData 함수와 $fetch 함수를 사용하여 수행할 수 있습니다.

useFetch 함수를 useAsyncData 함수와 $fetch 함수로 바꾸면 아래와 같은 코드가 됩니다.


<script setup>
const { data: posts, error } = await useAsyncData('posts', () =>
  $fetch('https://jsonplaceholder.typicode.com/posts/')
);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

useFetch 중에서는 $fetch가 이용되지만 useAsyncData에서는 $fetch를 이용하지 않고 다른 함수로 변경할 수 있습니다.useFetch와 달리 두 번째 인수의 함수 중 처리를 추가할 수도 있습니다.즉 useFetch보다 useAsyncData 를 이용하는 것으로 복잡한 처리를 실시할 수 있게 됩니다.


const { data: posts, error } = await useAsyncData('posts', () => {
  console.log('fetch'); //追加の処理
  return $fetch('https://jsonplaceholder.typicode.com/posts/');
});

useAsyncData 의 첫 번째 인수에는 캐시에 이용되는 유니크 키를 설정하고 있습니다.키는 생략이 가능하며 생략 시 자동으로 키가 설정됩니다.이 경우의 키는 파일명과 행번호를 이용하여 작성됩니다.


<script setup>
const { data: posts, error } = await useAsyncData(() =>
  $fetch('https://jsonplaceholder.typicode.com/posts/')
);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

posts라고 하는 키를 설정하고 취득한 값은 useNuxtApp().payload.data 에서 접근할 수 있습니다.


<script setup>
const { data: posts, error } = await useAsyncData(() =>
  $fetch('https://jsonplaceholder.typicode.com/posts/')
);
console.log(useNuxtApp().payload.data);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">{{ post.title }}</li>
    </ul>
  </div>
</template>

posts라는 키에 100개의 데이터가 포함되어 있는 것을 알 수 있습니다.

키를 지정하지 않은 경우도 동작 확인하면 자동으로 키가 설정되어 있는 것도 확인할 수 있습니다.

$fetch에 대하여

useAsyncData 중 갑자기 나온 $fetch 함수에 대해 확인해 두겠습니다.

$fetch는 도우미 함수로 내부에서는 ofetch 라이브러리를 이용하고 있습니다. ofetch 라이브러리는 Node, 브라우저, 워커 상에서 이용할 수 있는 fetch API입니다.

fetch 함수라면 익숙한 사람도 있을 것이므로 fetch 함수와의 대표적인 차이를 확인해 두겠습니다.

fetch 함수에서는 취득한 데이터를 JSON Parse 할 필요(response.json())가 있지만 ofetch에서는 자동으로 수행해 줍니다.


//$fetch関数
const posts = ref([]);
const data = await $fetch('https://jsonplaceholder.typicode.com/posts/');
posts.value = data;

//fetch関数
const posts = ref([]);
const response = await fetch('https://jsonplaceholder.typicode.com/posts/');
const data = await response.json();
posts.value = data;

fetch에서 POST 요청을 실행할 때 JSON.stringify에서 JSON 형식의 데이터로 변환하는데 ofetch에서는 자동으로 수행해 줍니다.


//$fetch関数
const data = await $fetch('/api/user', {
  method: 'POST',
  body: { name: 'John Doe' },
});

//fetch関数
const response = await fetch('/api/user', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({ name: 'John Doe' }),
});

그 외에는 존재하지 않는 URL에 접속했을 경우 response.ok이 false일 경우 에러 처리를 해 주거나 interceptors나 autoretry나 base URL을 설정할 수도 있습니다.

useLazyFetch

useFetch와 useLazyFetch의 차이점을 확인하기 위해 useFetch의 코드를 useLazyFetch의 코드로 다시 씁니다.


<script setup>
const { data: posts, error } = await useLazyFetch(
  'https://jsonplaceholder.typicode.com/posts/'
);
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <ul>
      <li v-for="post in posts" :key="post.id">
        <NuxtLink :to="`/posts/${post.id}`">{{ post.title }}</NuxtLink>
      </li>
    </ul>
  </div>
</template>

먼저 "/"에 접속하여 상단 링크에서 /posts로 이동합니다.현재 설정에서는 useFetch에서 useLazyFetch로 변경해도 차이는 알 수 없습니다.

다름을 명확히 하기 위해 JSONPlaceHolder로부터의 데이터 취득 시간이 지연되도록 브라우저의 디벨로퍼 툴에서 네트워크 속도를 "Slow 3G"로 변경합니다.

다시 동작확인을 하면 useFetch 의 경우는 /posts 에 접속하면 바로 posts 페이지의 화면이 표시되는 것이 아니라 잠시 시간이 지나면 "Post 일람" 이라고 취득한 데이터가 동시에 표시됩니다.useLazyFetch의 경우는 처음에 "Post 일람"이 표시되고 나서 그 후 취득한 데이터가 표시됩니다.

문서를 확인하면 "Bydefault, useFetch blocks navigation untilits asynchandler is resolved."라고 나와 있고, useFetch가 asynchandler가 resolve 될 때까지 네비게이션을 차단하고 있기 때문에 데이터 취득이 완료될 때까지 페이지로의 이동이 이루어지지 않습니다. useLazyFetch에서는 네비게이션이 차단되지 않으므로 페이지로 이동하여 "Post 일람"이 데이터 취득이 완료되기 전에 나타나게 됩니다.

useFetch 옵션의 lazy를 true로 설정하면 useLazyFetch와 같은 동작이 됩니다.


const { data: posts, error } = await useFetch(
  'https://jsonplaceholder.typicode.com/posts/',
  {
    lazy: true,
  }
);

useLazyFetch에서 되돌리는 pending을 이용하여 데이터 취득 중에만 로딩을 표시할 수 있습니다.


<script setup>
const {
  data: posts,
  error,
  pending,
} = await useLazyFetch('https://jsonplaceholder.typicode.com/posts/');
</script>
<template>
  <div>
    <h1>Posts一覧</h1>
    <p v-if="error">{{ error }}</p>
    <p v-if="pending">Loading...</p>
    <ul>
      <li v-for="post in posts" :key="post.id">
        <NuxtLink :to="`/posts/${post.id}`">{{ post.title }}</NuxtLink>
      </li>
    </ul>
  </div>
</template>

브라우저 상에서 확인하면 데이터 취득이 완료되어 표시할 때까지 "Loading…" 메시지가 표시됩니다.

useFetch에서 pending을 설정해도 데이터 취득이 완료된 후 페이지가 표시되기 때문에 값이 항상 false이기 때문에 Loading…이 표시하게 하지는 않습니다.

첫번째 "/"에서 /posts로의 이동시에는 Loading…만 화면이 표시되지만 두번째부터는 취득한 캐시의 데이터가 이용되기 때문에 "loading…"으로 취득한 데이터가 표시됩니다.

useLazyFetch 함수는 useLazyAsyncData 함수와 $fetch 함수를 이용하기 편리하게 만든 래퍼입니다.따라서 useLazyFetch 함수와 동일한 처리를 useLazyAsyncData 함수와 $fetch 함수로 수행할 수 있습니다.

useState

컴포넌트 간이나 페이지 간에 상태 관리(데이터 공유)를 원하는 경우에 composables 의 useState 를 이용할 수 있습니다.useState는 심플한 상태 관리에 이용할 수 있으며 보다 복잡한 상태 관리라면 라이브러리의 Pinia를 이용하게 됩니다.

상태 관리란 어떤 것인지 이해하기 위해 ref 함수를 이용한 경우와 비교합니다.

about.vue 파일을 이용하여 ref 함수로 counter를 정의합니다.버튼에 의해 counter의 수를 늘릴 수 있도록 설정을 실시합니다.


<template>
  <div>
    <Head>
      <Title>Aboutページ</Title>
      <Meta name="description" content="Aboutページ" />
    </Head>
    <h1>About Page</h1>
    <h2>Counter</h2>
    <p>Count: {{ counter }}</p>
    <div><button @click="counter++">+</button></div>
  </div>
</template>
<script setup>
const counter = ref(0);
</script>

버튼을 클릭하면 Count의 수가 클릭만큼 늘어납니다.

Count를 늘린 후 상단 링크를 이용하여 다른 페이지로 이동하여 다시 about 페이지로 돌아오십시오.브라우저 상의 Count 값이 클리어되어 0으로 표시됩니다.이와 같이 ref함수에서는 페이지 간 이동을 하면 값이 클리어되어 유지할 수 없습니다.

페이지를 이동하더라도 count 값을 유지하고 싶을 때 useState를 이용할 수 있습니다.


<script setup>
const counter = useState('counter', () => 0);
</script>
counter는 ref 함수이므로 script 태그 내에서 counter 값을 확인하고 싶다면 counter.value로 접근할 수 있습니다.

useState의 첫 번째 인수에 key의 counter를 설정하여 초기값으로 함수로 0을 되돌리고 있습니다.key를 설정하여 나중에 다른 페이지에서 공유한 counter에 액세스합니다.

버튼을 클릭하여 다른 페이지로 이동한 후 다시 about 페이지로 돌아오십시오.앞의 ref함수와는 달리 count의 값이 유지되고 있음을 확인할 수 있습니다.

한 페이지 내에서는 페이지를 이동해도 useState를 통해 상태를 유지할 수 있음을 알 수 있었습니다.

다음은 다른 페이지에서도 counter 값에 접근할 수 있는지 확인합니다.users.vue 파일을 사용하여 방금 설정한 키의 counter와 useState를 이용합니다.


<template>
  <div>
    <h1>Usersページ</h1>
    <p>Count:{{ counter }}</p>
    <NuxtPage />
  </div>
</template>
<script setup>
const counter = useState('counter');
</script>

about 페이지에서 버튼을 클릭하여 count를 늘린 후에 /users/list 페이지로 링크에서 이동합니다.사용자 리스트 페이지에서도 count의 값을 유지할 수 있는 것을 확인할 수 있습니다.useState를 통해 상태 유지와 컴포넌트 간의 데이터 공유를 수행할 수 있음을 알 수 있었습니다.

링크를 통해 about 페이지에서 이동하지 말고 /users/list 페이지에 직접 접속해 주세요./users/list 페이지를 연 상태에서 브라우저 새로고침에도 문제가 없습니다.

브라우저에서 확인하면 count 값이 들어가지 않은 상태에서 브라우저 상에 표시됩니다.이때 값은 undefined입니다.정의가 되어 있지 않기 때문에 표시되어 있지 않습니다.

이후 링크를 이용하여 about 페이지로 이동하고 돌아오면 Counter 값이 표시됩니다.이와 같이 useState를 이용한 경우 정의하고 있는 컴포넌트로 먼저 상태 초기화를 해야 합니다.

export function useCounter() {

}

에러 핸들링

여기에서는 Nuxt3에 있어서의 에러 핸들링의 설정에 대해 확인합니다.이 장을 다 읽으면 NuxtErrorBoundary, 도우미 함수 createError, showError, useError, clearError, 에러페이지error.vue, hooks 등의 이해를 높이고 문서의 Error handling 설명도 이해할 수 있을 것입니다.

NuxtErrorBoundary

NuxtErrorBoundary는 클라이언트 측에서 발생하는 오류를 포착하여 미리 설정한 오류 내용을 브라우저 상에 보여주고 싶을 때 이용할 수 있는 컴포넌트입니다.어떻게 설정하는지 확인해보겠습니다.

users.vue 파일에는 이하의 내용을 기술합니다.


<template>
  <div>
    <h1>Usersページ</h1>
    <NuxtPage />
  </div>
</template>

users 디렉토리의 list.vue 파일을 이용하여 클라이언트 측에서 의도적으로 오류를 발생시키는 코드를 추가하기 위해 먼저 정상 동작하는 코드를 추가합니다.

ref 함수에서 counter를 정의하고 버튼을 추가합니다.클릭하면 counter 값이 늘어나는 버튼을 추가합니다.브라우저에서 /users/list 에 접속하여 버튼을 클릭하면 count 의 수가 늘어납니다.


<template>
  <div>
    <h1>ユーザリスト</h1>
    <h2>Counter</h2>
    <p>Count: {{ counter }}</p>
    <div><button @click="inc">+</button></div>
  </div>
</template>
<script setup>
const counter = ref(0);
const inc = () => {
  counter.value++;
};
</script>

아까 코드에서 문제없이 동작하는 것을 확인하고 이번에는 의도적으로 오류를 발생시키기 위해 도우미 함수 createError를 throw시킵니다.헬퍼 함수 createError는 Nuxt에서 제공하는 함수로 클라이언트 측에서도 서버 측에서도 이용할 수 있으며 오류 객체를 생성할 수 있습니다.


create Error 에 대해서는 나중에 좀 더 자세히 설명하고 있습니다.

<script setup>
const counter = ref(0);
const inc = () => {
  throw createError('エラー発生');
  counter.value++;
};
</script>

버튼을 클릭하면 도우미 함수 create Error로 인한 오류 객체가 throw되고 브라우저 콘솔에 오류 메시지가 표시됩니다.이때 화면에는 아무런 변화가 없습니다(오류는 브라우저 상에 표시되지 않습니다).

그런 다음 list.vue 파일의 부모 컴포넌트인 users.vue 파일에서 NextPage 컴포넌트를 NuxtErrorBoundary 컴포넌트로 랩합니다.


<template>
  <div>
    <h1>Usersページ</h1>
    <NuxtErrorBoundary>
      <NuxtPage />
    </NuxtErrorBoundary>
  </div>
</template>

다시 버튼을 클릭하여 오류를 발생시킵니다.이전과는 달리 throw된 오류가 NuxtErrorBoundary에 의해 잡히기 때문에 콘솔에는 아무런 메시지가 표시되지 않습니다.또한 NuxtErrorBoundary에서 랩한 컴포넌트의 내용(list.vue 파일의 내용)도 브라우저 상에 나타나지 않게 됩니다.

NuxtErrorBoundary의 이름이 붙은 Slot을 이용하여 원래 표시되어야 할 내용(NuxtPage)과는 다른 오류 내용을 표시할 수 있습니다.


<template>
  <div>
    <h1>Usersページ</h1>
    <NuxtErrorBoundary>
      <NuxtPage />
      <template #error="{ error }">
        <p>{{ error }}</p>
      </template>
    </NuxtErrorBoundary>
  </div>
</template>

이와 같이 NuxtErrorBoundary와 template 태그를 조합함으로써 오류가 발생한 경우에는 template 태그를 설정한 위치에 오류 내용이 표시됩니다.

브라우저 상에 표시되어 있는 에러는 지우고 싶은 경우에는 error 의 값을 null 로 설정해야 합니다.헬퍼함수 clearError도 존재하지만 클라이언트측에서 이용해도 error의 값을 클리어 할 수 없기 때문에 error.value 에서 값을 null 로 합니다.

헬퍼 함수 clear Error는 추후 이용하겠습니다.

<template>
  <div>
    <h1>Usersページ</h1>
    <NuxtErrorBoundary>
      <NuxtPage />
      <template #error="{ error }">
        <p>{{ error }}</p>
        <button @click="resetError(error)">Clear Error</button>
      </template>
    </NuxtErrorBoundary>
  </div>
</template>
<script setup>
const resetError = (error) => {
  error.value = null;
};
</script>

에러 발생 후에 표시되는 "Clear Error" 버튼을 클릭하면 에러 화면이 사라지고 처음에 표시되었던 카운트가 다시 표시됩니다.

여기서는 NuxtPage 태그를 NuxtErrorBoundary 태그로 랩했는데 일반 컴포넌트를 랩해서 이용할 수도 있습니다.

Nuxt 3 에 포함되는 Nitro 서버(내부에서는 h3 이용)를 이용하여 서버 사이드의 처리를 실시할 수 있습니다.Nitro 서버에 의해 클라이언트 측에서 액세스 가능한 API Route를 작성할 수 있습니다.서버로서 데이터베이스에의 접속도 실시할 수 있으므로 데이터베이스를 이용한 환경에서 데이터의 취득과 추가의 동작 확인도 실시합니다.

server

API Route 를 작성하기 위해 server 디렉토리를 작성하고 그 아래에 api 디렉토리를 작성합니다.처음에 클라이언트측에서 /api/hello 에 액세스 하면 "Hello World" 를 되돌리는 심플한 API Route 를 추가합니다.

api 디렉토리에 hello.ts 를 작성하여 이하의 코드를 기술합니다.hello.ts 파일에서는 defineEventHandler 함수를 이용합니다.


export default defineEventHandler(() => 'Hello World');

server/api/디렉토리에 hello.ts 파일을 작성했을 경우 클라이언트에서 액세스하는 URL은 /api/hello가 됩니다.

pages의 index.vue 파일에서 useFetch 함수를 이용하여 /api/hello에 접근합니다.


<script setup>
const { data } = useFetch('/api/hello');
</script>
<template>
  <div>
    <h1>Main Page</h1>
    <h2>{{ data }}</h2>
  </div>
</template>

브라우저에서 확인하면 hello.ts로 반환된 "Hello World"가 브라우저 상에 나타납니다.

데이터베이스 접속

Server API Route 에서는 데이터베이스로의 접속을 실시할 수 있습니다.데이터베이스는 SQLite 데이터베이스를 이용합니다.SQLite 데이터베이스에 직접 접속하는 것도 가능하지만 ORM 의 Prisma 를 이용해 실시합니다.Prisma에서는 데이터 모델을 정의함으로써 데이터베이스를 조작할 때 SQL이 아닌 객체 메서드를 이용할 수 있습니다.서버에서 데이터베이스에 액세스하는 경우는 반드시 Prisma를 경유하게 됩니다.

Prisma 설정

npm 명령을 이용하여 prisma 라이브러리를 설치합니다.


 % npm install prisma

Prisma용 설정 파일을 만들기 위해 npxprisma init 명령을 실행합니다.


 % npx prisma init

명령어를 실행하면 prisma 디렉토리가 생성되고 그 안에 Schema.prisma 파일이 생성됩니다.env 파일도 함께 작성됩니다.

Schema.prisma 파일은 Prisma 설정 파일로 기본적으로 PostgreSQL 데이터베이스 연결 설정이 이루어지고 있으므로 sqlite로 변경합니다.


generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "sqlite" //デフォルトではpostgresql
  url      = env("DATABASE_URL")
}

환경변수 DABABASE_URL은 .env 파일 안에서 정의되어 있으므로 PostgreSQL 설정에서 SQLite 설정으로 변경합니다.SQLite는 파일에 정보를 저장하므로 dev.db를 지정합니다.


DATABASE_URL="file:./dev.db"

SQLite는 릴레이션 데이터베이스이므로 보통 테이블 간 릴레이션을 설정하는데 이번에는 테이블 Task만 작성하므로 Task 모델을 prisma.schema 파일로 정의합니다.모델에서는 테이블의 열명이나 틀을 설정합니다.id 와 task, completed 로 구성되어 id 는 자동 설정되어 task 는 문자열(String), completede 는 Boolean(진위치)로 설정하였습니다.


generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}

model Task {
  id      Int      @id @default(autoincrement())
  task   String
  completed    Boolean
}

모델 정의가 완료되면 마이그레이션을 수행하면 DATABASE_URL로 지정하여 dev.db 파일이 prisma 디렉토리에 작성되고 Task 모델을 바탕으로 Task 테이블이 생성됩니다.


% npx prisma migrate dev

명령어를 실행하면 마이그레이션 이름을 묻기 때문에 임의의 이름을 붙여주세요.명령이 완료되면 SQLite 데이터베이스에 Task 테이블이 생성됩니다.

Prisma는 데이터베이스 관리 GUI의 Prisma Studio도 이용할 수 있으므로 npxprisma studio 명령으로 부팅합니다.


 % npx prisma studio

API Route 추가

API Route를 추가하기 위해 server/api/task.ts 파일을 만듭니다.task.ts 파일 중에서는 설치한 prisma/client에서 Prisma Client를 가져와 Prisma Client 인스턴스를 만들고 task 테이블에서 findMany 메서드로 전체 데이터를 가져옵니다.


import { PrismaClient } from '@prisma/client';

export default defineEventHandler(async () => {
  const prisma = new PrismaClient();
  const tasks = await prisma.task.findMany();

  return tasks;
});
데이터베이스에서 데이터를 가져올 때는 select문을 이용하는데 Prisma를 이용하고 있기 때문에 select문이 아닌 findmany라는 메서드를 이용하고 있습니다.

task.ts 파일 작성 후 클라이언트 측에서는 pages/index.vue 파일 중 useFetch를 이용하여 /api/task로 GET 요청을 보냅니다.GET 리퀘스트의 결과 되돌리는 데이터를 이용해 v-for 디렉토리로 전개하고 있습니다.브라우저 상에는 SQLite에서 가져온 Task가 표시됩니다.


<script setup>
const { data: tasks } = useFetch('/api/task');
</script>
<template>
  <div>
    <h1>Main Page</h1>
    <ul>
      <li v-for="task in tasks" :key="task.id">{{ task.task }}</li>
    </ul>
  </div>
</template>

입력 양식 추가

API Route를 통해 데이터베이스에서 데이터를 얻을 수 있었기 때문에 다음에는 데이터베이스에 데이터를 추가하는 방법을 확인하겠습니다.

새로 작업을 추가할 수 있도록 pages/index.vue 파일에 입력 양식을 추가합니다.


<script setup>
const task = ref('');
const { data: tasks } = useFetch('/api/task');

const addTask = () => {
  console.log(task.value);
  task.value = '';
};
</script>
<template>
  <div>
    <h1>Main Page</h1>
    <ul>
      <li v-for="task in tasks" :key="task.id">{{ task.task }}</li>
    </ul>
    <form @submit.prevent="addTask">
      <div>
        <input v-model="task" />
      </div>
      <div>
        <button type="submit">タスクを登録</button>
      </div>
    </form>
  </div>
</template>

ref 함수에서 reactive 변수 task를 정의하고 input 요소의 v-model로 지정합니다.버튼을 클릭하면 submit 이벤트에 의해 addTask 함수가 실행됩니다.

POST 요청 전송

addTask 함수 중 /api/task에 대해 POST 요청을 송신합니다.


const addTask = () => {
  const { data } = useFetch('/api/task', {
    method: 'post',
    body: { task: task.value },
  });
  task.value = '';
};

GET 요청, POST 요청을 API Route의 /api/task에서 식별하기 위해 event 객체를 이용합니다.event 객체의 node.req.method에 요청 종류가 포함되어 있습니다.


import { PrismaClient } from '@prisma/client';

export default defineEventHandler(async (event) => {
  console.log(event.node.req.method);
  const prisma = new PrismaClient();
  const tasks = await prisma.task.findMany();

  return tasks;
});

API Route 처리는 서버 측에서 실행되기 위해 npm run dev 명령을 실행한 터미널에 GET 요청에서는 "GET", POST 요청에서는 "POST"로 표시됩니다.

event.node.req.method 값에 요청의 종류가 들어있음을 알았기 때문에 그 값들을 이용하여 분기를 실시합니다.POST 요청의 내용은 readBody 함수에 event를 지정하면 얻을 수 있습니다.


import { PrismaClient } from '@prisma/client';

export default defineEventHandler(async (event) => {
  const prisma = new PrismaClient();
  if (event.node.req.method === 'GET') {
    const tasks = await prisma.task.findMany();

    return tasks;
  }
  if (event.node.req.method === 'POST') {
    const body = await readBody(event);
    return { body };
  }
});

브라우저 상에서 입력 양식에서 문자를 입력하고 "작업 등록" 버튼을 클릭하면 body 내용물이 반환됩니다.

서버 측에서는 body 안에는 {task:'입력한 문자열'}이 들어가 있기 때문에 body를 이용하여 데이터베이스에 작업을 추가합니다.create 메서드를 이용하여 task와 completed를 설정하고 있습니다.completed는 데이터 작성 시 작업이 완료되지 않아 false로 하고 있습니다.데이터베이스 측에서 초기값을 false로 하고 있는 경우는 completed 속성 설정이 필요하지 않습니다.


if (event.node.req.method === 'POST') {
  const body = await readBody(event);
  const newTask = await prisma.task.create({
    data: {
      task: body.task,
      completed: false,
    },
  });
  console.log(newTask);
  return newTask;
}

입력 양식에서 "작업 등록"을 클릭하면 데이터베이스에 추가할 수 있게 됩니다.데이터베이스 추가 후 브라우저 상에 추가한 데이터를 보여주기 위해 refresh 함수를 이용하면 /api/task에 대한 GET 요청이 실행되므로 즉시 추가한 데이터가 브라우저 상에 나타납니다.


const { data: tasks, refresh } = useFetch('/api/task');

const addTask = () => {
  const { data } = useFetch('/api/task', {
    method: 'post',
    body: { task: task.value },
  });
  refresh();
  task.value = '';
};

task.ts 파일 내에서 리퀘스트 메소드 분기를 이용해 처리를 나누었는데 HTTP 리퀘스트 마다 파일을 나눌 수도 있습니다.GET 요청 시 task.get.ts, POST 요청 시 task.post .ts, PUT 및 DELETE는 각각 task.put.ts, task.delete.ts입니다.

import { PrismaClient } from '@prisma/client';

task.posts.ts 파일을 아래와 같이 기술합니다.

예를 들어 모든 리퀘스트의 송신지의 URL 을 취득하고 싶은 경우에는 api 에 middleware 디렉토리를 작성하고 그 아래에 log.ts 파일을 작성한 후에 아래의 코드를 기술합니다.


export default defineEventHandler((event) =>
  console.log('New request: ' + event.node.req.url)
);

클라이언트에서 API의 엔드포인트로 요청을 보내면 nPM run dev 명령을 실행한 터미널에 접속한 URL이 표시됩니다.

렌더링 모드

Nuxt 3 에서는 렌더링 모드를 페이지마다 변경할 수 있습니다.

Nuxt에서는 렌더링 모드에는 CSR(Client Side Rendering), Universal Rendering, SSG(Static Site Genarators) 등이 있습니다.Nuxt 의 렌더링 모드의 디폴트는 Universal Mode 라고 불리며 CSR 와 SSR 를 조합하여 렌더링을 실시하고 있습니다.

상기 렌더링 외에 Nuxt3에서는 Hybrid Rendering 이 등장하여 Route Rules 를 nuxt.cofig.ts 내에서 설정함으로써 페이지마다 렌더링 모드를 변경할 수 있습니다.이 기능은 집필시 문서에 의하면 active development(Route rules are still under active development, and subject to change.)와 같으니 본 문서에서는 ssr 옵션만 이용하여 동작 확인합니다.


export default defineNuxtConfig({
  routeRules: {
    // Static page generated on-demand, revalidates in background
    '/blog/**': { swr: true },
    // Static page generated on-demand once
    '/articles/**': { static: true },
    // Set custom headers matching paths
    '/_nuxt/**': { headers: { 'cache-control': 's-maxage=0' } },
    // Render these routes with SPA
    '/admin/**': { ssr: false },
    // Add cors headers
    '/api/v1/**': { cors: true },
    // Add redirect headers
    '/old-page': { redirect: '/new-page' },
    '/old-page2': { redirect: { to: '/new-page', statusCode: 302 } }
  }
})
문서상에서 active development에서 production ready가 되는대로 다른 옵션도 동작확인을 할 예정입니다.

Universal Mode

Universal Mode는 기본 상태에서 /posts에 처음 접속했을 경우의 네트워크 탭을 확인하여 서버에서 어떤 데이터가 반환되었는지 확인합니다.

/posts에 접속하면 서버측에서 작성(SSR)된 HTML이 Responser Headers를 보면 Context-type의 text/html로 되돌려져 있음을 확인할 수 있습니다.이것으로부터 최초로 액세스했을 경우에는 SSR에 의해 서버측에서 HTML이 작성되어 클라이언트에 반환되고 있는 것을 알 수 있습니다.

페이지의 소스를 확인하면 <divid="_nuxt"> 요소 안에 Posts 정보가 포함되어 있습니다.

SSR에서는 서버측에서 작성한 HTML 를 받아 브라우저상에 표시하는 것 뿐만 아니라 클라이언트측에서도 서버측과 같은 처리를 실시합니다.클라이언트 측에서 실시하는 처리를 Hydration이라고 합니다.

Hydration이라는 단어는 이해하기 힘들겠지만 매우 간단하게 설명하자면 서버에서 받은 HTML 에 이벤트 청취자를 설정하고 인터랙티브한 페이지로 만드는 것입니다.SSR에서는 서버로부터 받은 HTML 을 표시하기 위해서 바로 페이지는 표시되지만 자바스크립트 설정이 완료되지 않았기 때문에 자바스크립트 처리가 완료될 때까지 인터랙티브한 조작을 수행할 수 없습니다.그 때문에 Hydration이라고 하는 처리가 이루어집니다.

서버측과 동일한 처리를 한다고 아까 말씀드렸지만 브라우저의 개발 도구 요소에서 script 태그를 보면 페이지를 작성하기 위해 필요한 Post 데이터도 HTML과 별도로 받고 있는 것을 확인할 수 있습니다.클라이언트 측에서도 페이지를 작성하기 위해서 필요한 Java Script 를 다운로드와 받은 데이터를 이용해 페이지를 작성하고 요소에 이벤트 청취자나 어플리케이션의 상태 등을 설정합니다.그런 다음 최종적으로 Java Script 처리가 완료된 인터랙티브 페이지로 업데이트됩니다.

다음에 /(루트)에 접속한 후에 /posts에 접속합니다.루트에 액세스하면 서버 측에서 렌더링된 페이지가 반환됩니다.

페이지를 이동했을 때 이동 후 페이지는 다시 서버에서 HTML을 받는지 또는 서버에서 받은 자바스크립트를 이용하여 페이지를 클라이언트 측에서 갱신하는지 확인하기 위해 네트워크 탭에 표시된 "Clear" 버튼을 이용하여 이력을 삭제합니다.

이력을 지우고 상단의 Posts 링크를 이용하여 /posts 로 이동합니다.처음에 /posts 한 경우와는 달리 SSR에서 생성된 HTML 데이터를 받는 것이 아니라 브라우저에서 직접 JSONPlaceHolder에 접속하여 데이터를 취득하고 있는 것을 알 수 있습니다.클라이언트 측에서 처리를 하고 있으므로 CSR(Client Side Rendering)이라고 불립니다.

이와 같이 디폴트 Universal Mode는 SSR과 CSR을 조합하여 렌더링이 이루어지는 것을 알 수 있습니다.

CSR Mode

Universal Mode의 동작을 이해할 수 있었기 때문에 다음은 CSR 모드로 변경합니다.렌더링 모드 변경은 nuxt.config.ts 파일로 진행합니다.페이지마다 설정을 실시할 수 있으므로/posts 이하의 페이지에서 ssr의 설정을 false로 합니다.false로 하는 것으로 CSR 모드가 됩니다.


// https://nuxt.com/docs/api/configuration/nuxt-config
export default defineNuxtConfig({
  routeRules: {
    '/posts/**': { ssr: false },
  },
});

설정 변경 후 /posts 에 액세스를 실시합니다.

아까와 마찬가지로 서버에서 되돌리는 페이지의 내용을 네트워크 탭에서 봐도 아무것도 표시되어 있지 않습니다.

페이지의 소스를 봐도 <divid="_nuxt"></div>가 있을 뿐 Post에 관한 데이터는 없습니다.

Posts의 데이터는 브라우저에서 직접 JSONPlaceHolder에 액세스하여 가져옵니다.

Universal Mode와 CSR Mode의 차이를 이해할 수 있었습니다.
